핵심 해석
이번 실패는 “센서가 안 와서 멈춤”이 아니라, 센서는 정상이고 planner 출력도 있는데 safety가 emergency stop으로 계속 끊는 구조였습니다. 현업에서는 이런 경우를 planner 품질 이슈와 safety-planner coordination 이슈로 분류합니다.

무엇이 좋았는가

1. 통합 경로는 살아 있다

  • PX4 / MAVROS / ROS2 / Gazebo Classic / LiDAR topic 체인은 살아 있습니다.
  • FCU connected, OFFBOARD, armed 상태가 유지됐습니다.
  • 즉 실패가 “부팅 실패”나 “연결 끊김”이 아니라는 점이 중요합니다.

2. Safety 계층은 일을 제대로 했다

  • 최근접 장애물이 1.0m 안으로 들어오면 실제로 safe_cmd=0로 자르고 있습니다.
  • 현업 기준으로 safety monitor가 planner보다 상위에서 최종 제어를 보수적으로 잡고 있다는 점은 바람직합니다.
  • 즉 지금은 “안전하게 실패하는 시스템”에 가깝습니다.

3. 로그/기록 체계는 강하다

  • metrics.csv, events.log, summary.json, ros_states timeline/report가 모두 남습니다.
  • 실패 원인을 사람 감이 아니라 수치와 이벤트 시퀀스로 읽을 수 있습니다.
  • 논문화/실험관리 관점에서는 매우 큰 장점입니다.

무엇이 문제였는가

Planner는 계속 “가라”고 말한다

ros_states timeline에서 planner speed는 대부분 0.60~0.75m/s 수준을 유지합니다. 즉 planner는 좁은 틈을 여전히 통과 가능하다고 보고 있습니다.

Safety는 “지금은 안 된다”고 끊는다

같은 timeline에서 safe speed는 대부분 0.0m/s이고 이벤트는 emergency_stop_obstacle입니다. 이는 planner와 safety가 동일 공간을 다르게 해석한다는 뜻입니다.

가장 늦게 피하고 있다

현업 관점 판정

지금 상태는 corridor obstacle avoidance demo를 “보여주는 PoC” 단계는 넘었지만, 좁은 통로를 통과하는 controller/planner 성능 검증 단계로 가기엔 아직 이릅니다.

수정 REQ

아래는 현업 기준 우선순위입니다. 이미 일부는 현재 코드에 들어가기 시작했지만, 아직 “충분히 검증된 해결” 단계는 아닙니다.

P0. Planner-Safety 협조 구조 개선

  • gap target을 goal 쪽 edge가 아니라 center/committed target 위주로 고정
  • 여러 장애물 동시 inflation
  • clearance를 평균이 아니라 보수적 percentile/min 기반으로 유지
  • escape 진입을 더 빠르게 시작

P0. 맵 난이도 단계화

  • easy / medium / hard corridor 맵을 분리
  • 현재는 planner 검증과 map 스트레스 테스트가 섞여 있습니다
  • 회피 알고리즘 변경 시 난이도별 회귀 테스트가 가능해야 합니다

P1. Planner 디버그 토픽 추가

  • 선택된 gap start/end/center/target angle publish
  • target clearance, escape mode, bubble count publish
  • ros_states에서 내부 의사결정을 바로 보게 만들기

P1. Recovery / Hysteresis

  • normal ↔ emergency_stop 토글을 줄이는 hysteresis 필요
  • 지속적으로 blocked일 때 side-commit recovery 로직 추가
  • 현재는 chattering이 길게 이어집니다

P2. 논문용 비교 실험 구조

  • 현재 gap planner를 baseline으로 고정
  • 향후 VFH+/MPPI와 동일 맵 세트에서 비교
  • 성공률, 최소 장애물 거리, intervention count, runtime로 표준화

P2. 영상 자산 규칙 정리

  • 이번 평가용으로는 저장소 루트에서 mp4/mov/webm을 찾지 못했습니다
  • 이후부터는 run_id와 연결된 영상 파일명 규칙을 정해야 합니다
  • 예: artifacts/<run_id>/replay.mp4
