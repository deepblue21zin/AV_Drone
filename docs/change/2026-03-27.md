- 개념/용어, 로그 읽는 법, 장애 히스토리, 현재 상태, 긴 코스 시각화, 현업 평가 점수를 최신 상태로 다시 정리했다.

## 이유
- 사용자는 단순 우회가 아니라 사진처럼 기둥 사이를 통과하는 회피 경로를 원했고, reactive planner가 바깥으로 도망가지 않도록 코스 형상을 corridor형으로 다시 설계할 필요가 있었다.
- 브라우저에서 디버깅하던 상태를 나중에 그대로 재검토하고, 에러 분석 보고서와 artifact를 더 촘촘하게 연결할 필요가 있었다.
- 사용자는 JSON 원본보다 사람이 읽는 보고서와 그래프를 원했고, “무엇이 저장되는지 / 무엇은 자동이 아닌지”를 더 명확히 설명할 필요가 있었다.
- 현재 런타임은 정상인데, `goal reached` 이후 hover 상태를 센싱 실패처럼 오해하기 쉬웠다.
- README가 예전 실행 흐름과 섞여 보여 실제 실행 순서와 대시보드 URL을 한 번에 보기 어려웠다.
- obstacle demo에서 goal tolerance가 커서 path 검증이 빨리 끝나는 느낌이 있었다.
- 회피 동작은 확인됐기 때문에, 이제는 더 오래 주행하면서 여러 차례 회피 결정을 검증할 수 있는 코스가 필요했다.
- 사용자가 현재 코드와 운영 구조를 이해하기 어렵다고 느꼈기 때문에, 프로젝트 설명 HTML을 “대학생 2학년이 읽는 교과서” 수준으로 다시 써야 했다.
- 기존 `sim` 컨테이너가 빈 `HEADLESS`와 자동 `gzclient` 경로를 타면서 검은/빈 Gazebo 창이 반복되어, 서버는 headless로 띄우고 검증된 helper client만 붙이는 구조가 필요했다.
- 추가로 원인을 더 좁혀보니, 실제 컨테이너는 이미지 안의 오래된 `/opt/PX4-Autopilot/docker/sim/entrypoint.sh`를 실행하고 있어서, 워크스페이스에서 고친 Gazebo DISPLAY 복원 로직이 live 런타임에 전혀 반영되지 않고 있었다.

## 검증
- world 2종과 scenario/README/HTML 설명을 모두 같은 corridor pole layout으로 맞췄다.
- `ros_states` backend/frontend 코드에 debug recording 경로와 API를 연결했다.
- ros_states debug report generator 모듈을 추가했고, backend 문법 체크를 통과했다.
- `ros_states` API 기준 현재 상태가 `HOVER_AT_GOAL | goal reached`임을 확인했다.
- `README.md`, `drone1_autonomy.yaml`, `scenario_single_drone_obstacle_demo.yaml`를 함께 맞췄다.
- Classic/Gazebo용 world와 scenario obstacle manifest를 같은 배치로 맞췄다.
- `project_command_center.html`에 주요 섹션과 인터랙션이 정상 존재하는지 확인했다.
- `sim` 컨테이너 env에 `HEADLESS=1`, `GAZEBO_IP=127.0.0.1`, `GAZEBO_MASTER_URI=http://127.0.0.1:11345`가 실제로 반영됐는지 확인했다.
- helper로 띄운 `gzclient` log에 `Connected to gazebo master @ http://127.0.0.1:11345`와 `Publicized address: 127.0.0.1`가 기록되는 것을 확인했다.
- `gzserver` log에서 `Rendering will be disabled`가 원인이었음을 확인했고, `DISPLAY`를 복원한 뒤 `HEADLESS=1`만 유지하는 구조로 수정했다.
- `docker-compose.yml`의 `sim` command를 `/workspace/AV_Drone/docker/sim/entrypoint.sh`로 바꿔, 앞으로는 워크스페이스에서 수정한 최신 entrypoint가 실제 컨테이너에 바로 반영되게 했다.
- 수정 후 live `gzserver` 환경에 `DISPLAY=:0`가 실제로 들어가고, server log에서 `Can't open display` / `Rendering will be disabled` / `Unable to create CameraSensor`가 사라진 것을 확인했다.

## 남은 이슈
- world가 다시 바뀌었기 때문에 `sim` 컨테이너 재시작이 필요하다.
- goal과 코스 설명도 함께 바뀌었기 때문에 autonomy launch를 다시 띄워야 한다.
- 새 기능을 실제 UI에서 쓰려면 `ros_states` 패키지를 다시 빌드하고 런치 재시작해야 한다.
- 아직 실제 디버깅 세션이 저장되기 전까지는 `artifacts/_ros_states_debug/` 아래 report 파일이 없다.
- 완전한 reset 없이 이미 떠 있는 런타임에는 기존 goal 파라미터가 남아 있을 수 있다.
- 문서 내용은 최신 기준으로 갱신됐지만, 긴 코스의 실제 비행 결과는 sim 재시작 후 다시 검증해야 한다.
