AV_Drone를 완전히 이해하기 위한 프로젝트 설명서

이 페이지는 ROS 2, Gazebo, PX4, MAVROS, 드론을 처음 보는 사람이 지금 프로젝트가 어떻게 돌아가는지, 내가 어떤 코드를 왜 만들었는지, 로그를 어떻게 읽는지, 어떤 에러를 겪었고 어떻게 고쳤는지를 대학생 2학년 수준에서도 따라갈 수 있게 풀어쓴 프로젝트 설명서다. 목표는 “코드를 외우는 것”이 아니라 시스템이 어떤 입력을 받고 어떤 판단을 해서 어떤 출력을 내는지를 머리에 그릴 수 있게 만드는 것이다.

이 페이지를 어떻게 읽으면 좋은가

처음부터 코드로 들어가면 어렵다. 아래 순서대로 보면 프로젝트가 훨씬 쉽게 읽힌다.

1. 먼저 이 3줄만 이해

  • Gazebo Classic이 센서와 물리 환경을 만든다.
  • ROS 2 노드들이 scan, pose, state를 받아 속도 명령을 계산한다.
  • PX4는 그 명령을 따라가고, metrics logger가 결과를 artifact로 저장한다.

2. 그다음 봐야 할 파일

  • single_drone_autonomy.launch.py: 누가 같이 뜨는지
  • drone1_autonomy.yaml: 어떤 값으로 도는지
  • local_planner_node.py: 회피를 어떻게 계산하는지
  • autonomy_manager_node.py: 실제 비행 상태 머신이 어떻게 움직이는지

3. 실행 후 확인 포인트

  • /drone1/scan: 라이다가 살아 있는가
  • /mavros/local_position/pose: 위치가 들어오는가
  • /drone1/mission/phase: 현재 상태 머신이 어디까지 갔는가
  • artifacts/<run_id>/summary.json: 실험 결과가 남았는가

초심자가 가장 많이 헷갈리는 포인트

  • Gazebo를 켠다와 ROS 노드를 띄운다는 다른 단계다.
  • 노드가 떠 있다와 데이터가 정상적으로 흐른다도 다르다.
  • goal reached는 센싱 실패가 아니라, 목표 도달 후 hover 상태일 수도 있다.
  • reactive planner는 장애물을 보고 즉시 피하는 방식이지, 전체 맵을 보고 길을 찾는 global planner는 아니다.

현재 프로젝트를 한 문장씩 요약하면

  • 환경: Docker 안에서 PX4 SITL과 ROS 2 Humble을 분리 실행한다.
  • 센서: Gazebo Classic 라이다가 /drone1/scan을 만든다.
  • 인지: perception 노드는 가장 가까운 장애물 거리 하나를 계산한다.
  • 계획: planner는 goal 벡터와 회피 벡터를 합쳐 속도를 만든다.
  • 안전: safety monitor는 timeout이나 너무 가까운 장애물을 보면 정지시킨다.
  • 제어: autonomy manager는 OFFBOARD, arm, takeoff, hover를 관리한다.
  • 기록: metrics logger가 실험 증거를 artifact로 저장한다.

시스템 구조를 그림처럼 이해하기

아래 구조만 이해해도 이 프로젝트가 어떤 컴퓨터에서 어떤 정보가 만들어지고, 어디로 넘어가는지 머릿속에 그릴 수 있다.

1. Host

프로젝트 폴더, 터미널, 브라우저, GUI를 담당한다. 사람은 여기서 명령을 친다.

  • 프로젝트 경로: AV_Drone/
  • 브라우저: ros_states 열기
  • 필요하면 Gazebo GUI 보기

2. sim 컨테이너

PX4 SITL + Gazebo Classic이 실제 비행 환경과 센서를 만든다.

  • world: obstacle_demo.world
  • 라이다 모델: rplidar/model.sdf
  • 출력: pose, state, scan

3. ros 컨테이너

ROS 2 노드들이 센서와 상태를 받아 판단을 내린다.

  • perception
  • planning
  • safety
  • control
  • metrics

4. ros_states / artifacts

사람이 이해하기 쉽게 상태를 보여주고 실험 결과를 남긴다.

  • 상태 URL: http://localhost:5050
  • artifact: summary.json, metrics.csv
  • debug record: artifacts/_ros_states_debug/
  • 실험 장부: experiments/

토픽 기준 정보 흐름

  • /drone1/scan : 라이다 원시 거리 배열
  • /mavros/local_position/pose : 드론 현재 위치
  • /drone1/perception/nearest_obstacle_distance : 가장 가까운 장애물 거리
  • /drone1/autonomy/cmd_vel : planner가 만든 원본 속도 명령
  • /drone1/safety/cmd_vel : safety를 통과한 속도 명령
  • /mavros/setpoint_velocity/cmd_vel : PX4가 실제로 받는 최종 setpoint

패키지 기준 역할 분담

  • drone_bringup : 런치와 파라미터
  • drone_perception : scan을 사람이 쓰기 쉬운 거리 정보로 바꾸기
  • drone_planning : 어디로 얼마나 움직일지 계산
  • drone_safety : 위험하면 멈추기
  • drone_control : OFFBOARD, arm, takeoff, hover
  • drone_metrics : 로그/실험 결과 저장

        • docker compose logs -f sim : PX4/Gazebo 준비 여부 확인

        • scan_timeout, pose_timeout : safety monitor가 데이터를 못 받아 정지

        한 번 실행할 때마다 artifacts/<run_id>/가 생긴다. 이게 “실험 증거 묶음”이다.

        metadata.json실행 환경, git 상태, 시나리오, planner 버전나중에 재현하기 위해 필요

        3. ros_states 디버깅 기록

        http://localhost:5050에서 보는 디버깅 상태를 그대로 저장할 수 있다. 여기서 “브라우저에서 뭘 보고 있었는지”란 브라우저 픽셀 화면이 아니라, ros_states가 화면에 뿌리던 backend 상태를 뜻한다.

        • Save Snapshot : 현재 상태를 즉시 JSON 스냅샷으로 저장하고 report를 갱신
        • Start Recording : 지정한 초 간격으로 타임라인 기록 시작
        • Stop Recording : 기록 종료 후 마지막 전체 스냅샷 저장
        • Generate Report : 저장된 JSON/timeline을 읽어서 요약 + 시각화 + 그래프로 HTML 보고서 생성
        • Open Report : 가장 최근 생성된 report를 새 탭에서 열기
        • 저장 위치: artifacts/_ros_states_debug/<timestamp>_drone1_recording/

        남는 파일: session_manifest.json, timeline.jsonl, snapshots/, report.html, report_summary.json

        아무 버튼도 누르지 않으면 계속 자동 저장되지는 않는다. 즉, “지금 보고 있는 디버깅 상황을 남기고 싶다”면 Save Snapshot 또는 Start Recording을 먼저 눌러야 한다.

        4. 실험 장부와 변경 이력

        • experiments/ : 여러 런을 표 형태로 모으는 곳
        • docs/change/ : 오늘 무엇을 왜 바꿨는지 기록하는 곳
        • docs/error/ : 장애 보고서 HTML

        현업 관점에서는 코드 변경, 실행 결과, 문제 원인이 분리돼 있어야 나중에 분석이 편하다.

        5. 무엇을 계속 기록해야 하나

        • run metadata: commit, branch, scenario, world, planner version
        • raw time series: pose, scan, obstacle, cmd, phase
        • event log: timeout, arm, mode change, goal reached
        • outcome KPI: 성공/실패, 최소 장애물 거리, 소요 시간
        • issue/fix ledger: 무슨 문제였고 어떻게 고쳤는지

        • 해석: 이 상태는 “에러”가 아니라 목표 도달 후 hover일 수 있다.
        • 주의: goal_latch_enabled=true라서 한 번 goal을 잡으면 계속 true로 남을 수 있다.
        • 주의: launch를 여러 번 띄우면 같은 토픽에 publisher가 중복되어 상태가 혼란스러워질 수 있다.
        • 현재 pending: 긴 코스 버전은 sim 재시작 후 다시 검증해야 한다.
        실험 재시작 체크: world 파일이 바뀌었으므로 docker compose restart sim 후, autonomy launch를 다시 실행해야 긴 코스가 실제 반영된다.

        이제는 “회피가 아예 없다”가 아니라, “reactive avoidance는 있으니 그 다음 무엇을 해야 하나”라는 관점으로 backlog를 다시 정리해야 한다.

        REQ-01. 4대 드론 namespace / launch 구조

        • 완료 기준: /drone1 ~ /drone4 토픽 충돌 없음
        • 완료 기준: 4대 모두 개별 arm/takeoff/goal 제어 가능
        • 완료 기준: metrics/artifact가 기체별로 구분 저장

        REQ-02. Reactive를 넘어서는 planner

        • 현재 상태: LiDAR reactive avoidance 있음
        • 다음 목표: global/map-aware planner 또는 MPPI 기반 local-global 결합
        • 완료 기준: 복잡한 장애물 배치에서도 정지하지 않고 우회 성공률 확보

        REQ-03. Health monitor와 danger state

        • 상태: NORMAL / DEGRADED / DANGER / FAIL
        • 센서 손실, FCU disconnect, battery, collision risk를 통합 판단
        • 완료 기준: danger 이유가 topic/service로 외부에 공개됨

        REQ-04. Task model과 재할당 엔진

        • task 정의: waypoint, area scan 등
        • 오류 기체의 orphan task 추출
        • 남은 기체에게 cost 기반 재할당

        REQ-05. Safe return / MPPI 복귀

        • 위험 기체는 그냥 멈추지 않고 복귀 경로를 따라 이탈
        • 복귀 기체는 hazard bubble을 broadcast
        • 완료 기준: return success rate와 secondary collision count 측정

        REQ-06. 실험 자동화와 회귀 검증

        • scenario regression suite
        • pass/fail 자동 판정
        • run_id와 figure/table 자동 연결

        논문으로 발전시키는 방향

        사용자 아이디어인 “위험 드론은 빠지고, 다른 드론이 task를 이어받고, 위험 드론은 MPPI로 복귀”는 충분히 좋은 논문 주제가 된다. 다만 회피만이 아니라 mission continuation까지 묶어야 메시지가 강해진다.

        추천 논문 제목 방향

        Failure-Aware Multi-UAV Mission Continuation with Risk Broadcast, Dynamic Task Reallocation, and Safe-Return MPPI

        핵심 기여 포인트

        • 위험 감지
        • 위험 정보 공유
        • task 재할당
        • MPPI 복귀
        • mission continuation 평가

        왜 강한 주제인가

        단순 장애물 회피보다 임무 지속성과 회복력을 이야기하기 때문에 시스템 논문 메시지가 더 강해진다.
