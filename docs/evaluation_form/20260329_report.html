시스템 로직 한눈 정리

현재 baseline은 “센서 → local planner → safety clamp → autonomy manager → PX4 setpoint” 구조입니다. 즉 planner가 직접 PX4를 건드리는 게 아니라, 반드시 safety와 autonomy manager를 거쳐 최종 setpoint로 나갑니다.

Gazebo Classic + PX4

시뮬레이터와 FCU 상태를 생성합니다. 라이다 scan과 pose의 원천입니다.

MAVROS / ROS2 Topic

/mavros/state, /mavros/local_position/pose, /drone1/scan이 여기로 들어옵니다.

Gap-based Local Planner

LiDAR에서 열린 gap을 찾고 /drone1/autonomy/cmd_vel을 만듭니다.

Safety Monitor

장애물이 너무 가깝거나 scan/pose/cmd가 timeout이면 속도를 0으로 자릅니다.

Autonomy Manager

OFFBOARD/ARM, TAKEOFF, FOLLOW_PLAN, HOVER_AT_GOAL phase를 관리하고 최종 MAVROS setpoint를 publish합니다.

컴포넌트별 코드 로직

Autonomy Manager

파일: src/drone_control/drone_control/autonomy_manager_node.py

  • WAIT_STREAM → OFFBOARD_ARM → TAKEOFF → HOVER_AFTER_TAKEOFF → FOLLOW_PLAN → HOVER_AT_GOAL state machine을 가집니다.
  • Safety가 통과시킨 safe_cmd를 받아 MAVROS setpoint로 보냅니다.
  • 즉 planner가 좋아도 safety에서 끊기면 실제 FCU setpoint는 0에 가까워집니다.

Gap-based Local Planner

파일: src/drone_planning/drone_planning/local_planner_node.py

  • LiDAR를 전방 search sector로 자른 뒤 free mask를 만들고, bubble을 적용해 장애물 주변을 막습니다.
  • 후보 gap들을 점수화한 뒤 target angle을 선택합니다.
  • 현재 HEAD에서는 다중 bubble, gap center bias, target commitment, conservative clearance percentile이 추가되었습니다.

Safety Monitor

파일: src/drone_safety/drone_safety/safety_monitor_node.py

  • Pose timeout, scan timeout, planner cmd timeout, emergency stop distance를 감시합니다.
  • 현재 파라미터 기준 emergency_stop_distance = 1.0m입니다.
  • last_scan_min <= 1.0이면 무조건 safe_cmd = 0입니다.

Lidar Obstacle

파일: src/drone_perception/drone_perception/lidar_obstacle_node.py

  • LaserScan의 valid range 중 최소값만 골라 /drone1/perception/nearest_obstacle_distance로 publish합니다.
  • ros_states의 “Nearest Obstacle” 카드가 이 값을 읽습니다.

Metrics Logger

파일: src/drone_metrics/drone_metrics/metrics_logger_node.py

  • 실행마다 run artifact 디렉터리를 만들고 metrics.csv, events.log, summary.json를 남깁니다.
  • git commit/branch/dirty, planner name/version, seed, scenario manifest 경로도 같이 남깁니다.
  • 논문 실험 장부의 기반이 되는 계층입니다.

ros_states Recorder

파일: src/ros_states/ros_states/ros_monitor.py

  • _ros_states_debug 아래에 session, snapshots, timeline.jsonl, report.html을 만듭니다.

  • 사람이 브라우저에서 보던 상태를 구조화된 데이터로 남긴다는 점이 핵심입니다.

런타임 phase와 이번 실패의 실제 흐름

정상 기대 흐름

  1. WAIT_STREAM: pose/state/scan을 기다립니다.
  2. OFFBOARD_ARM: OFFBOARD 전환과 arm 요청을 보냅니다.
  3. TAKEOFF: 목표 고도까지 상승합니다.
  4. FOLLOW_PLAN: planner가 수평 cmd를 만들고 safety가 검증합니다.
  5. HOVER_AT_GOAL: goal reached 후 정지합니다.

이번 런의 실제 흐름

  1. 부팅과 takeoff는 정상적으로 넘어갔습니다.
  2. 문제는 FOLLOW_PLAN에 들어간 뒤 시작됐습니다.
  3. planner는 지속적으로 0.6~0.7m/s 수준 명령을 만들었습니다.
  4. 하지만 nearest obstacle이 1.0m 이하로 반복 진입해 safety가 수천 번 zero clamp를 걸었습니다.
  5. 결국 mission phase는 끝까지 FOLLOW_PLAN에 머물렀고 goal에 도달하지 못했습니다.

이번 수치가 말해주는 것

  • Planner output이 0이 아닌데 safe speed가 0이라는 것은 planner-safety coordination 문제입니다.
  • Pose/scan rate가 안정적인데 실패했다는 것은 인프라보다 planner 품질 쪽 이슈입니다.
  • 최근접 장애물이 0.20~0.22m에 오래 머무는 것은 “gap 진입을 늦게 하거나 너무 안쪽으로 붙는” 행동과 일치합니다.
  • 현재 코드 HEAD는 이 문제를 줄이기 위해 multi-bubble, center bias, commitment를 추가한 상태입니다.

평가 대상 런 vs 현재 코드(HEAD) 차이

평가 대상 런에서 보인 구조

  • Gap-based planner 방향은 맞았지만, 좁은 corridor에서 goal 욕심이 컸습니다.
  • 장애물 inflation이 충분치 않아 gap이 실제보다 넓게 보였을 가능성이 컸습니다.
  • Safety가 planner를 계속 끊는 장면이 recording과 metrics에서 동시에 관측됐습니다.

현재 HEAD에서 반영된 것

  • local_planner_node.py에 multi-bubble, center bias, commitment, percentile clearance, escape distance가 추가됐습니다.
  • drone1_autonomy.yaml에서 speed와 clearance 관련 파라미터가 더 보수적으로 조정됐습니다.
  • scenario_single_drone_obstacle_demo.yaml 기준 맵도 side pole + 큰 blocker 중심의 단순 corridor형으로 정리됐습니다.
