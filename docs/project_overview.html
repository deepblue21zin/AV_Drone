Architecture

현재 런타임 구조

왜 GUI를 host로 뺐나

Docker 안 Gazebo GUI는 AMD + Wayland 조합에서 반복적으로 blank gray window가 났다. 그래서 현재는 시뮬레이터 서버는 container에서 돌리고, GUI는 host에서 shared model cache를 통해 붙게 만들었다.

왜 Harmonic인가

Gazebo Classic은 레거시 계열이다. 지금 baseline은 이후 연구 확장을 위해 Gazebo Sim Harmonic을 기준으로 맞췄다. 구조가 현대적이고 장기 유지 관점에서 더 맞다.

Research Requirements

연구 주제를 달성하기 위한 REQ

REQ-01 Baseline Reproducibility

누가 clone / build / pull 해도 단일 드론 baseline이 다시 떠야 한다. 현재는 Docker + artifact + git context로 이 부분을 확보 중이다.

REQ-02 State Observability

pose, scan, phase, safety event, goal 상태가 기록되어야 한다. 그래야 failure-aware 연구로 확장했을 때 원인 분석이 가능하다.

REQ-03 Safe Flight State Machine

OFFBOARD, arm, takeoff, hover, follow plan, hold가 명확해야 한다. 연구 알고리즘은 이 안정된 상태 머신 위에 올라가야 한다.

REQ-04 Multi-UAV Namespace

drone1 기준 구조를 drone2, drone3, drone4로 복제 가능한 형태로 일반화해야 한다.

REQ-05 Failure Injection

특정 UAV를 failure state로 보내고, 임무 중단과 landing hazard 생성을 시뮬레이션할 수 있어야 한다.

REQ-06 Hazard Modeling

단순 충돌 회피가 아니라, failed UAV 주변의 landing bubble / descent corridor를 위험영역으로 반영해야 한다.

REQ-07 Task Reallocation

orphan task를 추출하고 남은 UAV에게 재할당하는 정책이 필요하다.

REQ-08 Quantitative Evaluation

latency, mission continuation success, secondary collision, recovery distance 같은 지표를 반복 실험으로 비교해야 한다.
