1. Thesis Direction

이 프로젝트를 어떤 논문 주제로 발전시킬 것인가

권장 주제 문장

단일 UAV가 failsafe로 이탈하는 상황에서, 남은 UAV들이 landing bubble/corridor를 위험영역으로 고려하면서 orphan task를 재할당받아 mission continuity를 유지하는 방법을 주제로 잡는 것이 가장 좋다.

즉 주제의 중심은 “MPPI를 썼다”가 아니라, fault-aware mission continuation system이다.

발표용 한 줄 메시지

현재는 단일 드론 기준선과 측정 체계를 확보했고, 이를 바탕으로 multi-UAV fault-aware mission continuation으로 확장하려는 단계다.

교수님께는 “이미 동작하는 플랫폼 위에 연구 질문을 올린다”는 점을 강조하는 편이 좋다.

Main Research Question

한 대의 UAV가 offboard-loss 혹은 failsafe로 이탈했을 때, 남은 UAV가 secondary collision 없이 제한 시간 안에 임무를 얼마나 잘 이어갈 수 있는가?

Sub Research Question

failed UAV 자체를 지우는 대신, 그 주변의 landing bubble / descent corridor를 위험영역으로 모델링하면 continuity와 safety가 실제로 좋아지는가?

Expected Contribution

재현 가능한 PX4/ROS 2 환경 위에서 fault-aware reallocation + hazard-aware replanning + traceable metrics 를 하나의 시스템으로 검증한다.

교수님 앞에서 강조할 포인트

  • 주제는 “드론 여러 대를 띄운다”가 아니라 “고장 후에도 임무를 지속하는가”다.
  • 기여는 새 수식 하나보다도, hazard-aware continuation architecture와 그 정량 검증 체계에 있다.
  • 현재 구현은 단일 드론이지만, 이미 Docker 기반 재현 환경, LiDAR 수신, phase/metrics logging까지 확보됐다.

6. Current Gaps

지금 무엇이 아직 논문 수준에 도달하지 못했는가

1. Multi-UAV 부재 현재는 single-UAV만 검증됐다. 논문 주제 본체는 4-UAV fault-aware continuation이므로 확장이 필수다.
2. Fault Injection 부재 고장 상황을 의도적으로 넣는 실험 체계가 아직 없다.
3. Hazard Model 부재 landing bubble / descent corridor를 아직 local planner cost에 반영하지 않았다.
4. Orphan Task 부재 failed UAV가 남긴 작업을 추출하고 재할당하는 상위 계획기가 없다.
5. 통계 검증 부재 반복 실험, baseline 비교, 신뢰구간/비모수 검정은 아직 시작 전이다.

8. Dynamic Simulation Target REQ

최종적으로 만들고 싶은 동적 시뮬레이션 목표 REQ

아래 REQ는 “현재 구현”이 아니라, 앞으로 논문/학회 제출 수준으로 도달하기 위해 만족해야 하는 목표 규격이다.

REQ-A. Mission Definition

  • UAV 4대가 waypoint inspection 또는 grid coverage 임무 수행
  • 각 task는 utility를 가짐
  • fault 후 완료된 utility 기준으로 Mission Continuity Score 계산

REQ-B. Failure Injection

  • 1대 UAV에 offboard-loss 또는 link-loss 삽입
  • failed UAV는 Land 또는 Hold failsafe 진입
  • fault time, faulted UAV ID, recovery latency 기록

REQ-C. Hazard Modeling

  • failed UAV 주변 bubble 반경 r_haz 정의
  • descent corridor / drift capsule 정의
  • remaining UAV local planner cost에 hazard 반영

REQ-D. Orphan Task

  • failed UAV의 미완료 task를 orphan set으로 추출
  • remaining UAV에 utility 기반 재할당
  • unfinished orphan ratio 계산

REQ-E. Local Replanning

  • remaining UAV는 hazard-aware MPPI 또는 동등한 local planner 사용
  • secondary collision 없이 path 수정
  • replanning latency와 p95 solve time 측정

REQ-F. Logging and Reproducibility

  • seed, scenario, fault time, task allocation, hazard parameter 저장
  • trajectory, event, KPI를 artifact와 CSV로 저장
  • 팀원이 같은 docker compose로 재현 가능해야 함

Baseline 비교 설계

방법설명의도
B0재할당 없음orphan task가 mission continuity를 얼마나 깎는지 보기 위함
B1재할당만, hazard 무시continuity는 좋아지지만 secondary collision이 늘어나는지 확인
P재할당 + bubble/corridor + hazard-aware replanning논문 본체

최종 논문화 조건

  • 4-UAV nominal이 안정적으로 돌아가야 한다.
  • fault injection이 반복 실험에서 재현 가능해야 한다.
  • baseline B0/B1/P를 같은 scenario, 같은 seed budget에서 비교해야 한다.
  • trajectory figure, KPI table, latency histogram, ablation 결과가 있어야 한다.
  • 현재 single-UAV artifact 구조를 multi-UAV log schema로 확장해야 한다.
