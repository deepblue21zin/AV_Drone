INC-001

ros_states 웹 백엔드 기동 실패

초기 치명 장애
증상브라우저에서 Activate 시 NetworkError
영향Mission Health 대시보드 사용 불가
상태해결 완료

대표 로그

ModuleNotFoundError: No module named 'flask'

문제 원인

  • ros 컨테이너 이미지에 python3-flask가 포함되지 않았습니다.
  • 그래서 ros2 launch ros_states ...가 떠도 백엔드 프로세스가 즉시 종료됐습니다.

수정 방법

  • docker/ros/Dockerfile에 python3-flask를 추가했습니다.
  • 런타임 컨테이너에도 Flask를 반영해 즉시 복구했습니다.

결과

  • ros_states가 정상 기동했습니다.
  • /api/activate 응답이 200 OK로 확인됐습니다.

ros_states JSON.parse 실패

데이터 직렬화 문제
증상Mission Health에 JSON.parse 에러
영향백엔드는 떠 있어도 대시보드 갱신 불가

Flight debug refresh failed:

수정 방법

결과

  • /api/flight_debug와 관련 API가 정상 JSON을 반환하게 됐습니다.
  • 브라우저 새로고침 후 Mission Health 렌더링이 복구됐습니다.

INC-004

LiDAR scan stale / obstacle metric 부재 / 회피 불능

핵심 런타임 장애
증상LiDAR Scan stale, No obstacle metric yet, planner 정지
영향회피 기동 불가, allow_motion_without_scan=false 때문에 의도적 정지
상태해결 완료

대표 로그 / 증거

ros2 topic echo /drone1/scan --once
-> timeout / 0 msg

sim container rclpy probe
-> SIM_SCAN_MSGS 1

ros container rclpy probe
-> ROS_SCAN_DONE 0

문제 원인

  • LiDAR plugin 자체보다 sim -> ros 컨테이너 사이의 ROS 2 DDS sample delivery 실패가 핵심 원인이었습니다.
  • 실행 중이던 컨테이너는 실제로 ipc=private 상태였고, 새 compose transport 설정이 런타임에 반영되지 않았습니다.

수정 방법

  • docker-compose.yml에 ipc: host를 적용했습니다.
  • ROS_DOMAIN_ID=0, ROS_LOCALHOST_ONLY=0, RMW_IMPLEMENTATION=rmw_fastrtps_cpp, FASTDDS_BUILTIN_TRANSPORTS=UDPv4를 두 컨테이너에 명시했습니다.
  • docker compose up -d --force-recreate sim ros로 새 설정을 실제 컨테이너에 반영했습니다.
  • 추가로 rplidar sensor model을 단순화하고 obstacle 높이를 5m로 올려 3m 비행고도에서도 scan plane과 교차하도록 맞췄습니다.

결과

  • ros 컨테이너에서 /drone1/scan 직접 수신이 성공했습니다.
  • ros_states /api/flight_debug에서 LiDAR, obstacle, planner, safe command가 모두 ok로 올라왔습니다.

  • parameter_snapshot.json, summary.json, config snapshot이 다시 채워졌습니다.

INC-008

Gazebo Classic GUI가 자동으로 뜨지만 빈 회색 창처럼 보임

GUI / X11 / camera 초기화 이슈
증상자동 Gazebo 창은 뜨지만 맵이 안 보이고 회색/빈 창처럼 보임
영향사용자는 Gazebo가 아예 안 켜진 것으로 오해하기 쉬움
상태해결 완료

대표 로그

Authorization required, but no authorization protocol specified
[Err] [RenderEngine.cc:749] Can't open display: :0
[Wrn] [GuiIface.cc:120] could not connect to display :0
Could not load the Qt platform plugin "xcb"

...

user_camera pose
x: -17.11
y: 15.29
z: 193.86

문제 원인

  • 자동 gzclient는 떴지만, 컨테이너가 호스트 X11 인증 파일(XAUTHORITY)을 보지 못해 DISPLAY=:0 연결에 실패했습니다.
  • 인증을 통과한 뒤에도 Gazebo Classic의 기본 orbit startup view가 장면 전체를 과하게 잡아, 카메라가 너무 멀리(z≈194) 튀어 회색 배경만 크게 보였습니다.
  • 즉 “GUI 미기동”이 아니라 X11 인증 실패 + startup camera auto-zoom 두 이슈가 겹친 상태였습니다.

수정 방법

  • docker-compose.yml에서 XAUTHORITY, XDG_RUNTIME_DIR를 컨테이너에 전달하고, 런타임 디렉터리를 read-only로 마운트했습니다.
  • docker/sim/entrypoint.sh에서 GUI 모드 시작 전 /root/.gazebo/gui.ini를 지워 stale GUI 상태를 초기화했습니다.
  • sim_assets/worlds/obstacle_demo.world의 기본 GUI camera를 orbit에서 fps로 바꾸고, corridor 중앙을 보는 pose(x=16, y=0, z=30)로 고정했습니다.
  • 자동 GUI 기본 경로는 유지하고, helper 스크립트는 fallback 용도로만 남겼습니다.

결과

  • 최신 sim 로그에서 Authorization required, Can't open display, Qt platform plugin 에러가 사라졌습니다.
  • 자동 프로세스가 다시 정상화됐습니다: gzserver ... obstacle_demo.world, gzclient --verbose.
  • live user_camera pose가 아래처럼 고정된 것을 확인했습니다.

현재 상태

복구된 항목

  • /drone1/scan이 ros 컨테이너에서 직접 수신됩니다.
  • /drone1/perception/nearest_obstacle_distance가 계산됩니다.
  • planner/safety command가 정상 토픽으로 publish됩니다.
  • metrics_logger가 다시 artifact를 기록합니다.
  • 기본 실행 경로에서 Gazebo Classic GUI가 자동으로 뜨고, corridor를 보는 startup camera가 적용됩니다.
  • ros_states URL은 http://localhost:5050입니다.

남은 관리 포인트

  • 연구 데모 품질 관점에서는 emergency stop threshold와 local reactive planner 파라미터 튜닝이 더 필요합니다.
  • run reset을 더 쉽게 만들기 위해 "한 번 클릭으로 clean restart" 경로를 만드는 것이 좋습니다.
  • 논문 단계로 가면 failure taxonomy와 rosbag2 raw recording도 추가하는 편이 좋습니다.
  • 자동 GUI는 복구됐지만, 호스트 세션이 바뀌면 먼저 xhost +SI:localuser:root, xhost +local:docker를 다시 열어주는 습관이 필요합니다.

운영 교훈

항목 교훈 실무 적용 포인트
증상 분리 UI 에러, transport 에러, planner 정지, mission complete를 섞어 보면 원인 파악이 느려집니다. 항상 state / pose / scan / obstacle / mission_phase를 같이 봅니다.
컨테이너 운영 compose 파일을 고친 것과 실행 중 컨테이너에 반영된 것은 다릅니다. 중요 transport 변경 후에는 --force-recreate가 필요합니다.
중복 launch ROS 2는 동일 노드명이 겹쳐도 얼핏 동작해 보여 더 위험합니다. 이상 동작 시 먼저 ros2 node list에서 duplicate부터 확인합니다.
문서 품질 README가 오래되면 실제 문제보다 운영 착오가 더 큰 장애를 만듭니다. active runtime, URL, restart 순서를 항상 최신 상태로 유지합니다.
GUI 진단 Gazebo 창이 빈 회색이라고 해서 항상 world 미기동은 아닙니다. X11 인증 로그와 live user_camera pose를 같이 보면 “미기동”과 “카메라 오배치”를 빠르게 구분할 수 있습니다.
