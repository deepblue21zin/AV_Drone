# 2026-03-26

## 변경 내용
- `local_planner_node.py`를 `nearest_obstacle` 기반 정지형 baseline에서 `LaserScan` 기반 reactive avoidance planner로 변경했다.
- planner가 `scan_topic`을 직접 구독해 전방 위험도, 좌/우 openness, repulsive vector를 계산하도록 바꿨다.
- `drone_planning/package.xml`에 `sensor_msgs` 의존성을 추가했다.
- `drone1_autonomy.yaml`에 회피 파라미터(`obstacle_influence_distance`, `obstacle_slow_distance`, `avoidance_gain`, `lateral_bias_gain` 등)를 추가하고 `allow_motion_without_scan` 기본값을 `false`로 바꿨다.
- `sim_assets/worlds/obstacle_demo.world`와 `sim_assets/gz/worlds/obstacle_demo.sdf`의 장애물을 경로를 가로지르는 불규칙 배치로 재구성했다.
- `scenario_single_drone_obstacle_demo.yaml`에 새 obstacle layout과 planner 성격을 반영했다.
- `single_drone_autonomy.launch.py`, `metrics_logger_node.py`, `docs/architecture.md`, `docs/project_command_center.html`의 planner 메타데이터/설명을 reactive avoidance 기준으로 갱신했다.

## 이유
- 기존 planner는 장애물을 발견하면 멈추기만 해서 사용자가 기대한 `회피 기동`이 나오지 않았다.
- 기존 obstacle 배치는 goal line 바깥쪽이라 회피 동작을 검증하기 어려웠다.
- 기록/문서가 옛 baseline 설명을 유지하면 실험 결과와 문서가 어긋나기 쉽다.

## 검증
- Python 문법 검사를 통해 planner/launch/metrics 파일이 import 가능한지 확인할 예정.
- world/scenario 파일은 정적 검토로 obstacle 배치가 goal line을 가로지르는지 확인했다.

## 남은 이슈
- 이 planner는 local reactive avoidance이며, global map 기반 우회나 corridor planning은 아직 없다.
- 실제 Gazebo Classic 런타임에서 파라미터 튜닝이 더 필요할 수 있다.
- `simulator_mavlink poll timeout` 같은 런타임 문제가 남아 있으면 회피 로직과 별개로 PX4/MAVROS 체인을 먼저 안정화해야 한다.

## 추가 디버깅 메모
- `ros_states` 기준으로 `FCU/pose`는 정상인데 `LiDAR scan`과 `nearest_obstacle`만 stale인 상태를 재현했다.
- `/drone1/scan --once`와 `/drone1/perception/nearest_obstacle_distance --once`가 둘 다 timeout되는 것을 확인했다.
- `sim` 컨테이너 안 Gazebo sensor model을 단순화해 `rplidar`에서 ROS ray sensor plugin만 남기고, obstacle 높이를 5m로 올려 3m 비행 고도에서 실제로 scan plane과 교차하도록 수정했다.
- 이 변경은 `sim` 재기동 후 반영된다.


## 추가 디버깅 메모 2
- 실제 원인은 LiDAR plugin 자체가 아니라 `sim -> ros` 컨테이너 사이의 ROS 2 DDS sample delivery 실패였다.
- 증거:
  - `sim` 컨테이너 내부 `rclpy` probe는 `/drone1/scan`을 정상 수신했다.
  - 같은 시점 `ros` 컨테이너 내부 `rclpy` probe는 `/drone1/scan`을 0건 수신했다.
  - 일반 테스트 토픽도 `ros -> sim` 사이에서 수신되지 않아 센서 전용 문제가 아니라 transport 계층 문제로 확인했다.
- 수정:
  - `docker-compose.yml`의 `sim`, `ros` 서비스에 `ipc: host`를 적용했다.
  - 두 서비스 모두 `ROS_DOMAIN_ID=0`, `ROS_LOCALHOST_ONLY=0`, `RMW_IMPLEMENTATION=rmw_fastrtps_cpp`, `FASTDDS_BUILTIN_TRANSPORTS=UDPv4`를 명시했다.
  - `docker compose up -d --force-recreate sim ros`로 컨테이너를 강제 재생성해 실제 런타임에 반영했다.
- 재검증:
  - 재생성 후 `ros` 컨테이너에서 `/drone1/scan` 직접 수신이 성공했다.
  - `ros_states /api/flight_debug`에서 `LiDAR Scan`, `Nearest Obstacle`, `Planner Cmd`, `Safety Cmd`가 모두 `ok`로 올라오는 것을 확인했다.
  - 같은 시점 mission summary는 `HOVER_AT_GOAL | goal reached | OFFBOARD | armed`였다.
- 추가 수정:
  - `metrics_logger_node.py`의 parameter snapshot 수집이 ROS 2 Humble에서 `list_parameters()` 호출로 죽는 문제를 확인했다.
  - `_write_parameter_snapshot()`를 `self._parameters.keys()` 기반으로 바꿔 Humble에서 죽지 않도록 수정했다.
- 주의:
  - `ros_states`는 백엔드가 재시작되면 브라우저 상태와 어긋날 수 있다. 이 경우 새로고침 후 `Activate`를 다시 눌러야 현재 상태가 반영된다.
